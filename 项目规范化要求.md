# 孤独症儿童辅助软件 Android 项目规范化要求

## 一、项目定位

本项目是一个面向孤独症儿童照护、训练与辅助沟通场景的 Android 应用。

项目特点：
- 页面数量较少
- 功能流程相对清晰
- 以本地静态数据 + 少量动态数据为主
- 当前阶段优先保证：结构清晰、页面可运行、便于后续继续迭代

因此，本项目采用轻量化的 **Android MVP + Jetpack Compose** 架构，不做过度设计，不引入不必要的复杂层级。

---

## 二、技术栈与总体要求

### 2.1 技术栈
- 开发语言：Kotlin
- UI：Jetpack Compose
- 架构：MVP
- 数据来源：
  - 静态初始数据：assets
  - 用户运行时产生的数据：应用内部存储
- 导航：统一由 navigation 模块管理

### 2.2 架构职责
- **Model**：负责数据模型与数据读取/保存
- **View**：Compose 页面，只负责展示和用户交互
- **Presenter**：负责业务逻辑、状态处理、调用 Repository
- **Repository**：统一负责读取 assets、本地文件，以及后续可能扩展的数据源

### 2.3 核心开发约束
- Composable 页面中**不能直接调用 Repository**
- 页面与业务逻辑解耦，页面只通过 Presenter 获取和更新数据
- 公共 UI 组件放在 `ui/components/`
- 仅在某个页面内使用的组件，放在该页面模块下的 `components/`
- 页面简单时，允许不单独拆 `components/`
- 单个 Screen 文件如果明显过长（建议接近或超过 300 行）再拆分组件，不要为了“规范”而强行拆太碎

## 四、页面模块组织规则

每个页面模块默认采用以下结构：

```text
presentation/xxx/
├── XxxContract.kt
├── XxxPresenter.kt
├── XxxScreen.kt
└── components/   # 可选
```

适用规则：

* **简单页面**：只有 `Contract + Presenter + Screen`
* **复杂页面**：再增加 `components/`
* **多个独立业务区域明显可拆**时再拆，例如：

  * 首页功能卡片区
  * 训练项目列表
  * 训练结果弹窗
  * 家长记录条目

### 命名规范

统一使用 PascalCase，例如：

* `HomeScreen.kt`
* `TrainingPresenter.kt`
* `RecordContract.kt`
* `ProfileCard.kt`

## 五、小程序迁移到 Android 的规范

### 5.1 页面迁移原则

原微信小程序中的每一个主要页面，在 Android 中对应一个 `Screen` 模块。

建议保持“小程序页面职责”与“Android 页面职责”一一对应，在迁移初期不要随意合并或大规模重构逻辑。

### 5.2 迁移时的代码落位

* 小程序中的页面展示结构 → `XxxScreen.kt`
* 小程序中的页面交互逻辑 → `XxxPresenter.kt`
* 小程序中的数据对象 → `data/model/`
* 小程序中的本地配置、静态题库、初始数据 → `assets/config/` 或 `assets/data/`

### 5.3 迁移时的设计要求

* 优先保证**功能等价**
* 不要求强行 1:1 还原小程序像素级 UI
* Android 端应更偏原生交互和 Compose 风格
* 页面少时，优先开发完整可运行版本，再做精细优化

## 六、数据规范

### 6.1 数据分层

本项目采用两层数据结构：

#### 第一层：静态数据（Assets）

位置：`app/src/main/assets/`

用途：

* 应用配置
* 训练题目/卡片
* 初始演示数据
* 页面默认内容
* 不需要运行时修改的数据

#### 第二层：动态数据（内部存储）

用途：

* 训练记录
* 任务完成状态
* 用户选择结果
* 简单本地偏好设置
* 家长填写的记录内容

### 6.2 数据读取优先级

统一遵循以下顺序：

```text
内存缓存 → 内部存储文件 → assets 初始数据
```

也就是：

1. 优先读取内存缓存
2. 若无缓存，则读取本地文件
3. 若本地文件不存在，则回退到 assets 中的初始数据

### 6.3 数据文件命名建议

命名应语义清晰，例如：

* `app_config.json`
* `training_items.json`
* `cards.json`
* `records.json`
* `user_profile.json`

禁止使用：

* `data1.json`
* `test.json`
* `new_data.json`

## 七、Model 设计规范

所有数据模型统一放在：

```text
data/model/
```

要求：

* 一个核心业务对象对应一个 Kotlin `data class`
* 字段命名清晰，不要使用拼音缩写
* 页面不复杂时，可直接：

  * `JSON -> data/model -> Presenter -> Screen`
* 如果后续页面出现明显的展示转换需求，再增加 UI 展示模型

### 当前推荐策略

由于项目页面较少、逻辑较轻，不建议一开始就拆出过多中间层模型。优先保持：

* 数据结构清晰
* 类型简单
* Presenter 易读

---

## 八、Repository 设计规范

所有数据读写统一通过 Repository 完成，位置：

```text
data/repository/
```

职责包括：

* 从 assets 中读取初始 JSON 数据
* 从本地内部存储读取动态数据
* 将运行时产生的数据写回本地文件
* 可选：维护内存缓存，减少重复读取

### 禁止事项

* Screen 直接读取 assets
* Screen 直接读写文件
* Presenter 跳过 Repository 直接操作文件系统

### 推荐命名

* `AppRepository.kt`
* `TrainingRepository.kt`
* `RecordRepository.kt`

如果当前项目体量很小，也可以先只有一个总仓库类：

* `DataRepository.kt`

---

## 九、字符串资源规范

所有**与具体数据无关的 UI 文本**统一放入：

```text
res/values/strings.xml
res/values-zh/strings.xml
```

适合放进去的内容：

* 页面标题
* 按钮文字
* 通用提示语
* 错误提示
* 加载文案
* 空状态文案

不建议放进去的内容：

* 训练题目正文
* 卡片内容
* 演示数据
* 与业务数据强绑定的长文本

这些更适合放在 `assets/data/` 中。

## 十、UI 组件规范

### 10.1 公共组件

放在：

```text
ui/components/
```

例如：

* `LoadingIndicator.kt`
* `ErrorView.kt`
* `EmptyView.kt`
* `PrimaryButton.kt`
* `SectionCard.kt`

要求：

* 无业务耦合
* 无页面专属逻辑
* 通过参数接收内容和回调

### 10.2 页面业务组件

放在：

```text
presentation/xxx/components/
```

例如：

* `TrainingItemCard.kt`
* `RecordListItem.kt`
* `ProfileHeader.kt`

要求：

* 只服务于对应模块
* 可以包含当前页面的展示逻辑
* 不要放进 `ui/components/` 里冒充通用组件

### 10.3 组件拆分原则

* 页面简单：直接写在 Screen 中
* 页面复杂：拆出业务组件
* 多页面复用：放到 `ui/components/`

---

## 十一、导航规范

统一在：

```text
navigation/
```

中管理页面路由。

要求：

* 页面路由集中定义
* 页面间跳转统一从导航层处理
* 简单返回上一级时，直接使用普通返回逻辑
* 不要提前写一套过重的导航封装

如果项目页面少，导航层只需要：

* 路由常量
* `NavGraph`
* 页面注册关系

---

## 十二、Presenter 与页面交互规范

标准数据流如下：

```text
用户操作 → Screen 回调 → Presenter 处理 → Repository 读写数据 → Screen 更新状态 → UI 重绘
```

要求：

* Screen 只接收状态和事件
* Presenter 负责输入校验、业务流程和错误处理
* Repository 负责实际数据读写
* 任何保存动作都必须通过 Repository
* 状态更新后由 Compose 自动重绘页面

---

## 十三、页面状态处理规范

每个页面至少应考虑以下状态：

* Loading
* Success
* Empty
* Error

如果页面很简单，也可以不强制封装成复杂的状态类，但至少要在 UI 中明确处理：

* 正在加载时显示什么
* 没有数据时显示什么
* 出错时显示什么

推荐优先复用公共组件：

* `LoadingIndicator`
* `ErrorView`
* `EmptyView`

## 十六、代码质量要求

### 必须遵守

* 页面代码可运行
* 不允许直接在 Screen 中操作数据文件
* 不允许把所有逻辑都塞进一个 Composable
* 不允许无意义地创建过多中间层
* 不允许出现明显临时代码命名（如 temp、test、demo_final2）

### 推荐遵守

* 关键函数写简要注释
* 公共逻辑抽到 utils 或 common 中
* 可复用 UI 抽成组件
* 页面代码保持清晰易读

---

## 十七、当前阶段开发原则

当前项目属于**小型、页面较少、从微信小程序迁移的 Android 项目**，开发时遵循以下优先级：

1. **先保证功能可运行**
2. **再保证目录结构清晰**
3. **再优化 UI 与交互细节**
4. **最后再考虑扩展性与复用性**

不要一开始为了“规范”而把项目设计得过重。
本阶段目标是：**用尽量清晰、适度规范、便于继续开发的结构，快速完成 Android 版本落地。**

---

## 十八、总结开发要求

请严格按照以下要求生成代码：

1. 使用 Kotlin + Jetpack Compose
2. 使用 MVP 架构
3. 所有页面逻辑通过 Presenter 管理
4. 所有数据读写通过 Repository 管理
5. 页面少，因此结构保持精简，不要过度拆分
6. 公共组件放在 `ui/components/`
7. 页面专属组件放在 `presentation/xxx/components/`
8. 静态数据放在 `assets/`
9. 动态数据保存到应用内部存储
10. 页面代码要可直接运行，避免生成只有结构没有实现的空壳代码
11. 如果某个页面较简单，可以不额外建立 `components/`
12. 如果某个 Screen 文件过长或 UI 结构明显复杂，再拆分组件
13. 不要在 Compose 页面中直接访问 Repository 或文件系统
14. 优先适配当前“小页面数量、从小程序迁移”的场景，不要照搬大型项目模板
15. 所有命名清晰、统一、语义化，避免临时命名和冗余抽象
